# Cache system (cargo-buckal)

This document describes the current cache behavior in `cargo-buckal` as implemented in the
Rust sources under `cargo-buckal/src/`.

## Overview

`cargo-buckal` keeps a small, persistent snapshot of the last Cargo dependency graph it saw.
The snapshot is used to compute a diff on the next run so it only updates BUCK files for
packages that actually changed.

The cache is intentionally simple:

- It records a fingerprint per `BuckalNode` (a unified dependency graph node).
- It is stored as a TOML file in the Buck2 repo root.
- It is versioned; stale versions are silently rebuilt and future versions cause an error.

## High-level flow

```
                    cargo metadata
                         |
                         v
              +--------------------+
              |   BuckalResolve    |     (graph of BuckalNode)
              +--------------------+
                    |          |
                    v          v
          +-----------+   +-----------+
          | new cache |   | old cache |     (buckal.snap on disk)
          +-----------+   +-----------+
                    \        /
                     v      v
               +---------------+
               |     diff()    |
               +---------------+
                      |
          +-----------+-----------+
          |           |           |
        Added      Changed    Removed
          |           |           |
          v           v           v
        generate    regenerate  clean up
        BUCK file   BUCK file   vendor dir
                      |
                      v
               save new cache
```

## Cache file location

The cache file is written to the Buck2 project root as:

```
<buck2-root>/buckal.snap
```

The Buck2 root is discovered at runtime via `buck2 root`.

## File format

The cache is serialized as pretty TOML with a generated header comment. The top-level
structure is:

- `version`: schema version (currently `3`).
- `fingerprints`: a `BTreeMap<PackageId, Fingerprint>` (sorted for deterministic output).

Each `fingerprint` is a 32-byte BLAKE3 digest, hex-encoded as a string.

Example (shape only):

```toml
# @generated by `cargo buckal`
# Not intended for manual editing.
version = 3

[fingerprints]
"path+file://($WORKSPACE)/crates/foo#foo@0.1.0" = "...hex..."
"registry+https://github.com/rust-lang/crates.io-index#serde@1.0.196" = "...hex..."
```

## Fingerprints

For each `BuckalNode` in the resolved dependency graph, the cache stores a
location-independent fingerprint:

```
fingerprint = BLAKE3(field-by-field hash of BuckalNode + sorted outgoing edges)
```

The fingerprint hashes individual fields selectively to ensure portability across
different checkout locations:

```
BuckalNode (node weight)
  ├── package_id         (canonicalized: workspace root → ($WORKSPACE) placeholder)
  ├── name, version, edition
  ├── features           (sorted for determinism)
  ├── kind               (FirstParty { relative_path } | ThirdParty)
  ├── targets[]          (name, kind, relative src_path, doctest, test)
  ├── source, links
  └── checksum

  Excluded: manifest_path (absolute; identity captured by kind + name/version/checksum)
  Normalized: targets[*].src_path (relativized against manifest_dir)
  Normalized: package_id (canonicalized via PackageIdExt)

BuckalDep (edge weight)
  ├── name              (may differ from package name if renamed)
  └── dep_kinds[]       (kind + optional platform constraint)
```

The fingerprint for each node hashes the fields above plus its sorted outgoing edges
(`BuckalDep` weights + canonicalized child `PackageId`). This means the cache changes
whenever *any* node field or dependency edge changes — not just features (as in the
previous version).

## Workspace canonicalization

To keep the cache portable across machines and directories, `PackageId` values that point
into the workspace are canonicalized via the `PackageIdExt` trait:

```
                  canonicalize()
    path+file:///home/user/myproject/...
              ────────────────────────>
    path+file://($WORKSPACE)/...
              <────────────────────────
                    resolve()
```

- `canonicalize(workspace_root)` — replaces the workspace root with `($WORKSPACE)`.
- `resolve(workspace_root)` — replaces `($WORKSPACE)` with the actual workspace root.

Canonicalization happens when building the cache; resolution happens when diffing.

## Versioning and invalidation

The cache schema is versioned via `CACHE_VERSION` in `cache.rs`.

- Current version: `3` (introduced for the graph refactor with `BuckalNode`-based fingerprinting).
- Version history:
  - `2`: Added multi-platform support.
  - `3`: Switched to `BuckalNode`-based fingerprinting (includes targets, source, checksum,
    edition, links, kind).

```
             load buckal.snap
                    |
                    v
          ┌─────────────────┐
          │ parse version   │
          └────────┬────────┘
                   |
        +----------+----------+
        |          |          |
   version < 3  version = 3  version > 3
        |          |          |
        v          v          v
   discard &     use it    error: upgrade
   rebuild                 Buckal
```

## Lifecycle in commands

### Construction

`BuckalCache::from_resolve(resolve, workspace_root)` builds a cache by iterating all nodes in
the `BuckalResolve` dependency graph, computing each node's fingerprint, and canonicalizing its `PackageId`.

### Read path

`get_last_cache(manifest_path)` (in `utils.rs`) attempts to load the cache. If loading fails
(missing, unparsable, or stale version), it falls back to rebuilding from current
`cargo metadata` via `BuckalResolve::from_metadata()`. If the rebuild itself fails (e.g.
Buck2 root not found, `cargo metadata` error, or a dependency cycle is detected), it logs a
warning and returns an empty cache — removal detection is disabled for that run.

### Write path

After applying updates, a new cache snapshot is always written via `cache.save()`.

### Commands that use the cache

```
cargo buckal migrate
  │
  ├── --no-cache or load fails?
  │     yes ──> BuckalCache::new_empty()
  │     no  ──> BuckalCache::load()
  │
  ├── BuckalCache::from_resolve()   (new cache from current metadata)
  ├── new_cache.diff(&old_cache)    (compute Added/Changed/Removed)
  ├── changes.apply(&ctx)           (generate/update/delete BUCK files)
  └── new_cache.save()

cargo buckal {add, update, remove}
  │
  ├── get_last_cache()              (load or rebuild)
  ├── run cargo {add, update, remove}
  ├── BuckalContext::new()          (re-read metadata after cargo command)
  ├── BuckalCache::from_resolve()
  ├── new_cache.diff(&old_cache)
  ├── changes.apply(&ctx)
  └── new_cache.save()
```

### Diff behavior

`new_cache.diff(&old_cache, workspace_root)` compares fingerprints:

- Present in new, missing in old: `Added`
- Present in old, missing in new: `Removed`
- Present in both but fingerprint differs: `Changed`

These changes drive BUCK generation and vendor directory cleanup. The diff resolves canonicalized
`PackageId` values back to their absolute workspace paths before returning.

## In-process cfg cache (platform mapping)

Separately from the on-disk cache, platform mapping uses an in-memory cache of
`rustc --print=cfg --target <triple>` results. This cache is per-process and initialized
once (via `OnceLock`) to avoid repeated `rustc` invocations when evaluating platform
predicates.

This cache is not persisted and does not affect `buckal.snap`.

## Troubleshooting notes

- If you see unexpected full regeneration, check whether `buckal.snap` is missing or has an
  older `version` value.
- If you move the workspace, the `($WORKSPACE)` placeholder allows the cache to remain valid.
- If you want to force a clean run, use `cargo buckal migrate --no-cache`.
- If you see "Cache was written by a newer version of Buckal", upgrade your `cargo-buckal`
  installation.
